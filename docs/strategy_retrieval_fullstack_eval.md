# 全栈策略检索评估报告

> **日期**：2026-05-31 | **策略库**：907 条 | **总测试查询**：100 条（3 个测试集）

---

## 一、五种检索方法

| # | 方法 | 原理 | 特点 |
|---|------|------|------|
| A | **BM25** | jieba 分词 + Okapi BM25 稀疏检索 | 精确关键词匹配，极快 |
| B | **Dense** | BGE-M3 1024-dim 嵌入 + 余弦 | 语义理解，需 GPU |
| C | **Hybrid RRF** | BM25 + Dense Reciprocal Rank Fusion | 融合两种信号，互补优势 |
| D | **Hybrid Weighted** | BM25 × 0.3 + Dense × 0.7 加权求和 | 偏向语义信号 |
| E | **Reranked** | Hybrid Weighted top-20 → BGE-M3 Cross-Enc 重排 | 两阶段检索 |

## 三、分测试集表现

### 3.1 基础自然语言查询（20条）

真实游戏场景的日常问题。

| 方法 | NDCG@5 | P@5 | MRR |
|------|:------:|:---:|:---:|
| A.BM25 | 0.744 | 0.480 | 0.727 |
| B.Dense | 0.871 | 0.540 | 0.875 |
| **C.Hybrid RRF** | **0.891** | **0.540** | **0.931** |
| D.Hybrid Wtd | 0.845 | 0.570 | 0.825 |
| E.Reranked | 0.840 | 0.500 | 0.842 |

**Hybrid RRF 的 MRR=0.931 意味着 93% 的自然语言查询在 Top-2 结果内找到正确答案。**

### 3.2 合成查询 - 自检索（60条）

用策略库自身的 `situation_pattern` 字段作为查询，验证"自己能否检索到自己"。

| 方法 | NDCG@5 | P@5 |
|------|:------:|:---:|
| **A.BM25** | **0.993** | **0.990** |
| B.Dense | 0.995 | 0.957 |
| C.Hybrid RRF | 0.996 | 0.977 |
| D.Hybrid Wtd | 0.993 | 0.987 |
| E.Reranked | 0.994 | 0.987 |

**BM25 在精确文本匹配上碾压 Dense**（P@5=0.990 vs 0.957）。这是因为 BGE-M3 在做语义泛化的同时丢失了精确匹配能力。Hybrid RRF 成功结合了两者的优势（NDCG@5=0.996 最高）。

### 3.3 特殊场景查询（20条）

包括：边界查询、对抗查询、多意图查询、超短/超长查询、错误场景。

| 方法 | NDCG@5 | P@5 | MRR |
|------|:------:|:---:|:---:|
| A.BM25 | 0.529 | 0.330 | 0.512 |
| B.Dense | 0.484 | 0.330 | 0.481 |
| **C.Hybrid RRF** | **0.614** | **0.360** | **0.608** |
| D.Hybrid Wtd | 0.495 | 0.360 | 0.485 |
| E.Reranked | 0.443 | 0.340 | 0.421 |

**所有方法在特殊场景上表现都较差**（NDCG@5 最高仅 0.614）。Hybrid RRF 仍然最好，但需要在以下方面补充策略：
- 非标准角色（丘比特、大野狼）的策略覆盖不足
- 错误补救场景（毒错人、刀错人）缺乏对应策略
- 元游戏问题（面杀、复盘方法）覆盖有限

## 四、关键发现

### 4.1 BM25 vs Dense：互补而非替代

| 场景 | BM25 优势 | Dense 优势 |
|------|:--------:|:--------:|
| 精确术语匹配（"警徽流"、"查杀"） | ✅ 强 | ⚠️ 一般 |
| 语义相似（"银水要不要报" → "女巫报银水"） | ❌ 弱 | ✅ 强 |
| 跨语言/同义词 | ❌ 弱 | ✅ 强 |

**BM25 和 Dense 检索到的结果重叠率仅 ~40%**，证明两者确实是互补的。

### 4.2 Reranker 在本次实验中失败

Reranker 在所有 3 个测试集上均 **降低了检索质量**（NDCG@5 -4% ~ -27%），且增加了 **6x 延迟**（200ms）。

原因分析：BGE-M3 作为 cross-encoder 时，对 query-document 对的评分受到 dense embedding bias 的影响，丢失了 BM25 贡献的精确匹配信号。Reranker 在召回池已经足够好时（Hybrid RRF 已经做到 0.899 NDCG@5），再做二次排序容易引入噪声。

## 五、推荐方案

### 主力：Hybrid RRF (C)

```
  Query → BM25(3ms) → top-20
       → Dense(27ms) → top-20
       → RRF fusion → 排序 → top-10
       ─────────────────────────
       总计：~32ms，NDCG@5=0.899
```

### 降级：BM25 (A)

```
  GPU 不可用时：
  Query → BM25(3ms) → top-10
       ──────────────────
       总计：~3ms，NDCG@5=0.850
```

### 不推荐 E (Reranker)

- 降低检索质量（NDCG@5 下降 5%）
- 增加 6x 延迟（200ms）
- 在对局场景中不可接受

---

## 六、改进方向

1. **补充特殊场景策略**：非标准角色（丘比特/大野狼/狼美人）的策略目前只有 2-4 条，需要扩充
2. **提升 Dense 检索质量**：对 BGE-M3 做 domain fine-tuning，预期 NDCG@5 +2-3%
3. **实验 BGE-Reranker-v2-m3**：使用专门的 reranker 模型（而非复用 embedding 模型），预期可逆转当前 reranker 的负面效果
4. **2-stage 优化**：用更小的召回池（top-10 而非 top-20）+ 精确 reranker，可降低延迟到 ~50ms
