# 团队组织与协作总览

## 一、团队构成

| 角色 | 人数 | 主要职责 |
|------|------|----------|
| 全栈开发 | 3 | 前端 / 后端 / Agent 都可能接手，按 issue/feature 横向认领 |
| AI 助手 | Claude / Codex / 等 | 在人类指挥下编码、调试、写测试，**最终代码与决策由人类负责** |

**项目周期**：2026-05-20 → 2026-06-10（21 天）→ 答辩 2026-06-11 ~ 06-20。

---

## 二、横向分工模型（重要）

**不按模块切人**，按 **feature/issue 切人**。同一人可能这周改前端，下周改 Agent。

### 这种模式的代价 & 我们的应对

| 代价 | 应对 |
|------|------|
| 没有"模块所有者"，代码风格容易漂移 | **规范文档（本目录）+ 严格 PR review** |
| 容易踩到别人改了一半的状态 | **每天上工前 `git pull --rebase origin main`，PR 小且快进** |
| 跨模块联调成本高 | **API/事件契约（`50-api-contract.md`）必须先改文档，再改代码** |
| 容易出现重复造轮子 | **接活前在群里报一下"我接 issue #N"，开 draft PR 占坑** |
| AI 改全栈时一致性差 | **每次让 AI 干活前显式喂规范文件路径（`70-ai-collaboration.md`）** |

### 必须遵守的协调动作

1. **认领任务**：在协作工具（Issue/飞书群/Trello 任选）报 owner，不抢、不偷接。
2. **开 draft PR 占坑**：动手后 24h 内推第一个 commit，避免别人重复劳动。
3. **跨模块改动事前通知**：动 `protocols/`、`engine/models.py`、`engine/phases.py` 这种"中枢文件" → 群里 @全员。
4. **冲突时谁先 merge 谁赢**：后来者 rebase 自己改。不强求顺序，避免互相等。

## 三、决策权矩阵

| 决策类型 | 决策方式 |
|----------|----------|
| 命名（变量/文件/接口名） | 改的人定，但要符合 `20`/`30`/`50` 命名规则 |
| 新增依赖（`requirements.txt`） | **全员 +1 才合并**，理由写 PR 描述 |
| 改动公开 API 字段 | **必须先改 `50-api-contract.md`，再实现** |
| 改动 `engine/models.py` 的 Enum | **必须先群里说一声**，影响所有 Agent/DB/前端 |
| 改动 Prompt 模板 | 改的人决定，但 LLM 行为差异显著时 PR 描述贴对比 |
| Bug 修复 / 文案 / 样式 | 改的人决定，1 人 review 通过即可 merge |
| 进阶方向选 B 还是 C | **群里集体决定后写进 `SKILLS.md`**，不要 PR 偷换 |

## 四、开发节奏（建议）

> 不强制，但偏离太多会拖慢答辩准备。

| 时间 | 阶段 | 关键交付 |
|------|------|----------|
| 第 1 周 | MVP 加固 + 联调 | 现有引擎 + Agent 跑通，PR review 流程跑顺 |
| 第 2 周 | 进阶方向冲刺 | B（评测）或 C（自进化）的核心实现 |
| 第 3 周 | UI 抛光 + 数据准备 | 答辩 Demo 流程跑顺，准备日志/Leaderboard |
| 答辩周 | 演练 + 修小 Bug | **冻结主干，只接 critical fix**，不接新 feature |

**冻结期定义**：答辩前 3 天（约 2026-06-08 起）进入冻结期，所有非关键 PR 推迟。

## 六、新成员（含新 AI）入职清单

## 七、AI 助手的特别说明

- **你不是模块所有者**，你的代码必须经过人类 review 才能进主干。
- **每次会话开始前**，主动列出你打算读的文件清单，让人类确认。
- **跨模块改动**前必须列出影响面，等人类点头再动手。
- **不要假设其他 AI 之前改了什么**——你看到的代码就是真相，不要凭印象。
- 详见 `70-ai-collaboration.md`。
